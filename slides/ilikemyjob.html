Data Pipeline

The data pipeline essentially takes data from produciton systems, cleans it and provides it in a format that can be queried. It is a lot like a water processing plant in your city, when it's working you won't even notice it is there, but it is still extremely important.

As stated before this data is what allows the BIAs to provide their services.

What are Datasets?

Bespoke information generated for a narrow purpose.

Advantages

  • Easy to make
  • Quick to deliver
  • Fits the intended purpose

An example of this is the survey data we retrieve from QILT.
It requires little more than parsing it and adding a couple of fields to it.

Disadvantages

  • Not always quick and easy
  • Maintenance
  • Stuck to a specific scope
  • Need to know all the columns at build time
  • Some datasets are quite complex. For instance, Ray created a dataset that has nearly 500 lines of code and 10 pages of documentation explaining it.
  • They require lots of upkeep. Made a dataset for Sathish and TM1, he is constantly asking for adjustments
  • Not very good at being used outside their intended purpose. For instace, TM1 datasets.
  • You need to know exactly what fields the end user requires.

The Data Warehouse

So what does it actually do?

Takes data from production systems, cleans it, applies business rules and provides the data in a format which allows cross-source analysis and encourages exploration.

Why not just use the source systems?

  • Designed for reading and writing
    • Slow for massive queries
  • At the wrong grain
  • Lacks dimensionality
  • Designed for fast reading and writing, however, only limited reading. Looking at 100 records simultaneously would be pushing it
  • The grain is at what level you want to see the information. For instance, we only care about a students daily movements, not anything finer. In all likelihood the grain is not going to match what you need for analysis
  • Dimensionality is what allows the facts to answer so many different questions, quickly. Want to know the students who are international? You could probably do that with the source data pretty easily. What about the international students who are enrolled in MA1301? Getting a bit slower. What about a count of all international students who enrolled in MA1301 split by year and study period? You've probably just brought the production system to its knees while the data warehouse hasn't broken a sweat.

The Process

  • Extract
  • Transform
  • Load

Extract

This is where we copy the data from the source systems to our database ready for processing. Happens nightly. Only slight transforms happen here. A common pattern is keeping a copy of the data after every step.

Transform

This slide betrays the complexity of what is actually happening here.
  • Each dot is a table which can be referenced in multiple files, each table is linked to the tables it consumes and also the tables it is consumed by.
  • Each step is creating intermediary tables which applies business rules and portioning.
  • Just as an aside, alot of our data is date portioned. Essentially we find stretches of time where a record hasn't updated and create a single portion for this date range. This stops us from having lots of duplicates where the data hasn't updated.

Load

Final stage is the load. In this part we take the results of all the business rules transformations and store it where the systems like Cognos and Tableu can access them.
Because loading the tables takes a slow process there would normally be an outage of a few minutes where Cognos could not work. We employ a renaming trick so the Cognos and Tableu never see an outage.

The move to Git

  • Code needs to be stored with all revisions
  • SVN was not good at this
  • Benefits of Git are:
    • Better history
    • Easier code reviews
    • Tighter integration with existing Atlassian ecosystem
    • Accountability for changes
  • It is important that code is not only stored, but the history of the code is also kept.
  • We were initially using SVN, but that was clunky and old. Downloading an enitre code repository consisted of download full copies of every branch someone had made
  • Git was an alternative that is used in many companies the world over.
  • It provides better history management, storing only the changes to files rather than the a full copy of the repository.
  • Atlassian offers a service for git called Bitbucket, which integrates nicely with both Confluence and Jira.
  • Git allows you to look at the person responsible for each line of code. This is critical for data governance

At the beginning of the year I pushed hard for the move to Git. It had been planned for quite some time, however, projects always kept coming up and blocking the way. Misty had created a Git methodoligies page in Confluence, and keeping those methods in mind I moved all four environments to Git.
